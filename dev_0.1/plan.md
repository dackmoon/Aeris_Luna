# Aeris / Luna 实现计划 v0.1

本文档记录 Luna 从当前形象设计到可运行自用 App 的阶段计划。`Aeris` 是长期项目/体系名称，`Luna` 是当前这个宝宝陪伴角色和 App 原型的具体名称。

当前主路线：先做混合 App，而不是纯网页或纯原生 App。App 外壳负责手机权限和设备能力，内部用 WebView 承载 Live2D、对话界面和口型同步逻辑。

## 总体路线

```text
形象方向确认
-> Android 混合 App 原型
-> Live2D 模型加载
-> TTS 播放和音频驱动口型
-> 麦克风录音和语音识别
-> 对话模型、人设和记忆
-> iOS / 树莓派扩展
```

## 技术选型建议

优先方案：

- App 框架：Capacitor / Ionic + React。
- UI 内核：WebView。
- 角色渲染：Live2D Cubism Web SDK 或成熟 Web Live2D 渲染库。
- 口型同步：Web Audio API 分析 TTS 音量，驱动 `ParamMouthOpenY`。
- Android 自用：打包 APK 后直接安装。
- iOS 自用：后续用 Xcode 真机安装或 TestFlight。

这个方案的好处是：录音、权限、本地文件等能力由 App 原生层处理，Live2D 和 UI 仍然沿用网页技术，后续也更容易迁移到树莓派 Chromium 全屏网页。

## 阶段 0：v0.1 形象与方向沉淀

目标：确认 Luna 的角色定位、视觉方向和第一批资产。

要做的事情：

- 完成 `character.md`：角色设定、说话风格、视觉气质。
- 完成 `prompts.md`：主形象、表情、视频参考提示词。
- 整理现有图片资产。
- 记录图片评审结果。
- 明确不再把图生视频作为主路线，只作为参考素材。

当前状态：

- 已有主形象方向。
- 已有 4 个基础表情图。
- 已确定 Live2D + App/WebView 为主路线。

## 阶段 1：v0.2 Android 混合 App 原型

目标：先在 Android 手机上跑起来一个最小 App。

要做的事情：

- 初始化 React + Capacitor 项目。
- 创建一个竖屏移动端界面。
- App 内部使用 WebView 展示角色区域和对话区域。
- 配置 Android 打包。
- 在 Android 手机上安装 APK。
- 验证 App 能正常启动、全屏显示、触摸交互正常。

暂不做：

- 真实对话模型。
- 真实语音识别。
- 长期记忆。
- 正式 Live2D 专属模型。

验收标准：

- Android 手机上可以打开 Luna App。
- 页面布局适合手机竖屏。
- 可以显示角色区域和基础聊天区域。
- 可以点击按钮触发测试事件。

## 阶段 2：v0.3 Live2D 模型加载

目标：在 App 的 WebView 中加载现成 Live2D 模型。

要做的事情：

- 找一个现成已绑定 Live2D 模型。
- 确认授权是否允许自用、网页展示和后续二次开发。
- 将模型文件放入项目资源目录。
- 在 WebView 中加载模型。
- 实现待机、眨眼、轻微呼吸。
- 尝试切换模型自带表情。

模型要求：

- 包含 `model3.json`。
- 有嘴巴参数，最好支持 `ParamMouthOpenY`。
- 模型不要太复杂，优先保证手机性能。

验收标准：

- Android App 中能显示 Live2D 模型。
- 模型动画流畅。
- 可以切换至少 2 个状态或表情。
- 长时间运行没有明显卡顿或发热异常。

## 阶段 3：v0.4 TTS 播放与口型同步

目标：让角色在播放语音时自动张嘴。

要做的事情：

- 准备本地测试音频，例如 `hello.mp3`。
- 在 App 中播放音频。
- 使用 Web Audio API 获取音频音量。
- 将音量映射到 Live2D 的 `ParamMouthOpenY`。
- 增加平滑算法，让嘴巴开合更自然。
- 音频结束后让嘴巴自然闭合。

核心逻辑：

```text
播放音频
-> 获取音量
-> 归一化为 0 到 1
-> 平滑处理
-> 写入 ParamMouthOpenY
```

验收标准：

- 点击按钮后能播放语音。
- 播放时嘴巴随声音开合。
- 静音时嘴巴闭合。
- 效果比图生视频更适合实时对话。

## 阶段 4：v0.5 麦克风录音与语音识别

目标：让用户可以用语音和 Luna 互动。

要做的事情：

- 申请 Android 麦克风权限。
- 实现按住说话或点击录音。
- 保存或上传录音。
- 接入语音识别服务。
- 将识别文本显示在聊天框。
- 根据识别内容触发 Luna 回复。

可选路线：

- 先使用云端语音识别。
- 后续再研究本地语音识别。

验收标准：

- 手机可以录音。
- 录音可以转成文字。
- 识别结果能进入对话流程。
- 失败时有明确提示。

## 阶段 5：v0.6 对话模型、人设和记忆

目标：让 Luna 开始具备基本陪伴对话能力。

要做的事情：

- 把 `character.md` 转成系统提示词。
- 建立基础人设配置文件。
- 接入一个对话模型 API。
- 根据用户输入生成回复。
- 回复时播放 TTS。
- 回复类型映射到 Live2D 表情状态。
- 建立简单记忆，例如宝宝昵称、兴趣、近期话题。

早期记忆可以使用：

- 本地 JSON。
- SQLite。
- 轻量 key-value 存储。

验收标准：

- Luna 能按人设回复。
- 回复语气温柔、简短、有陪伴感。
- 可以记住少量基础信息。
- 表情、语音、口型能配合对话状态。

## 阶段 6：v0.7 iOS 自用版本

目标：在 iPhone 上验证同一套 App。

要做的事情：

- 配置 iOS Capacitor 项目。
- 使用 Xcode 真机运行。
- 配置麦克风权限。
- 测试 WebView 中 Live2D 性能。
- 测试音频播放和口型同步。

注意事项：

- iOS 自用安装比 Android 麻烦。
- 免费 Apple ID 可能有签名时间限制。
- 长期自用可能需要 Apple Developer 账号或 TestFlight。

验收标准：

- iPhone 可以运行 Luna App。
- 麦克风和音频播放正常。
- Live2D 性能可接受。

## 阶段 7：v0.8 树莓派屏幕版本

目标：把 Luna 迁移到树莓派和自有屏幕上。

要做的事情：

- 准备树莓派系统和屏幕。
- 使用 Chromium 全屏打开 Web 版本。
- 测试 Live2D 性能。
- 连接麦克风和扬声器。
- 测试语音输入、TTS 输出和口型同步。
- 配置开机自启动。

验收标准：

- 树莓派开机后自动显示 Luna。
- 可以播放语音并驱动口型。
- 基础交互稳定。
- 长时间运行不会明显卡顿或崩溃。

## 当前下一步

最建议马上做：

1. 找一个现成已绑定 Live2D 模型。
2. 确认模型授权和参数。
3. 初始化 Android 混合 App 原型。
4. 先在手机上跑出一个空界面。
5. 再把 Live2D 模型加载进去。
