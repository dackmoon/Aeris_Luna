# Aeris / Luna 实现计划 v0.1

本文档记录 Luna 从当前形象设计到可运行自用 App 的阶段计划。`Aeris` 是长期项目/体系名称，`Luna` 是当前这个宝宝陪伴角色和 App 原型的具体名称。

当前主路线：先做混合 App，而不是纯网页或纯原生 App。App 外壳负责手机权限和设备能力，内部用 WebView 承载视频 loop 素材库、状态机、对话界面、TTS 播放和伪口型/说话状态逻辑。

## 总体路线

```text
形象方向确认
-> Android 混合 App 原型
-> 视频 loop 素材库
-> 状态机调度
-> TTS 播放和伪口型/说话状态
-> 麦克风录音和语音识别
-> 对话模型、人设和记忆
-> iOS / 树莓派扩展
-> Live2D 可选升级
```

## 技术选型建议

优先方案：

- App 框架：Capacitor / Ionic + React。
- UI 内核：WebView。
- 角色渲染：预生成视频 loop 素材库。
- 状态调度：前端状态机，根据用户操作和回复阶段切换视频池。
- 说话表现：TTS 播放时切换 `speaking` 视频池，必要时叠加轻微缩放、呼吸或嘴部贴片。
- Android 自用：打包 APK 后直接安装。
- iOS 自用：后续用 Xcode 真机安装或 TestFlight。

这个方案的好处是：不需要会画画、拆 PSD 或绑定 Live2D，早期实现成本低；录音、权限、本地文件等能力由 App 原生层处理，视频播放和 UI 仍然沿用网页技术，后续也更容易迁移到树莓派 Chromium 全屏网页。Live2D 保留为长期可选升级，不作为当前主路线。

## 阶段 0：v0.1 形象与方向沉淀

目标：确认 Luna 的角色定位、视觉方向和第一批资产。

要做的事情：

- 完成 `character.md`：角色设定、说话风格、视觉气质。
- 完成 `prompts.md`：主形象、表情、视频参考提示词。
- 整理现有图片资产。
- 记录图片评审结果。
- 明确短期主路线改为视频 loop 素材库 + 状态机 + TTS + 伪口型/说话状态。

当前状态：

- 已有主形象方向。
- 已有 4 个基础表情图。
- 已确定视频 loop + App/WebView 为短期主路线。
- Live2D 降级为长期可选升级。

## 阶段 1：v0.2 Android 混合 App 原型

目标：先在 Android 手机上跑起来一个最小 App。

要做的事情：

- 初始化 React + Capacitor 项目。
- 创建一个竖屏移动端界面。
- App 内部使用 WebView 展示角色区域和对话区域。
- 配置 Android 打包。
- 在 Android 手机上安装 APK。
- 验证 App 能正常启动、全屏显示、触摸交互正常。

暂不做：

- 真实对话模型。
- 真实语音识别。
- 长期记忆。
- 精准口型同步。
- 正式 Live2D 专属模型。

验收标准：

- Android 手机上可以打开 Luna App。
- 页面布局适合手机竖屏。
- 可以显示角色区域和基础聊天区域。
- 可以点击按钮触发测试事件。

## 阶段 2：v0.3 视频 loop 素材库

目标：建立 Luna 的基础动态视频素材库，用预生成视频表达角色状态。

要做的事情：

- 设计第一批状态：`idle`、`listening`、`thinking`、`speaking`、`happy`、`comforting`。
- 每个状态先准备 1 到 3 条短视频 loop。
- 视频时长建议 `3-8 秒`。
- 视频比例优先适配手机竖屏角色区域。
- 统一背景、构图、角色大小和画风。
- 将视频放入 `assets/video-references/` 或后续 App 资源目录。

素材要求：

- 动作轻微、循环自然。
- 角色脸部尽量稳定，不要漂移。
- 服装、发卡、胸前装饰尽量一致。
- `speaking` 不追求精确口型，只需要有说话感。
- 文件大小可控，避免手机加载过慢。

验收标准：

- 至少有 `idle`、`listening`、`thinking`、`speaking` 四类视频。
- App 中可以播放视频 loop。
- 不同状态切换时观感自然。
- 安卓手机上播放流畅。

## 阶段 3：v0.4 状态机调度

目标：根据用户操作和系统状态，自动选择合适的视频 loop。

要做的事情：

- 建立状态枚举：`idle`、`listening`、`thinking`、`speaking`、`happy`、`comforting`。
- 为每个状态建立视频池。
- 相同状态下随机选择视频，避免重复感。
- 设置视频结束后的续播规则。
- 处理状态切换，例如淡入淡出或直接切换。
- 先用按钮手动触发状态变化，后续再接真实对话流程。

核心逻辑：

```text
用户无操作 -> idle
用户开始说话 -> listening
等待模型回复 -> thinking
播放 Luna 回复 -> speaking
积极反馈 -> happy
安抚场景 -> comforting
```

验收标准：

- 点击按钮能切换不同状态。
- 每个状态能从对应视频池随机播放。
- 状态切换不卡顿。
- 角色表现有基础陪伴感。

## 阶段 4：v0.5 TTS 播放与伪口型/说话状态

目标：播放 Luna 的语音回复时，让角色进入“正在说话”的状态。

要做的事情：

- 准备本地测试音频，例如 `hello.mp3`。
- 在 App 中播放音频。
- 播放开始时切换到 `speaking` 视频池。
- 播放结束后回到 `idle` 或下一个状态。
- 可选：用 Web Audio API 获取音量，用音量控制轻微缩放、亮度、嘴部贴片或说话动效强度。
- 可选：准备多条 `speaking` 视频，随机调用，降低重复感。

核心逻辑：

```text
开始播放 TTS -> speaking
TTS 播放中 -> 随机 speaking loop / 轻微说话动效
TTS 结束 -> idle / listening
```

验收标准：

- 点击按钮后能播放语音。
- 播放时角色进入说话状态。
- 语音结束后能自然回到待机或倾听状态。
- 即使口型不精确，也能让用户感到 Luna 正在回应。

## 阶段 5：v0.6 麦克风录音与语音识别

目标：让用户可以用语音和 Luna 互动。

要做的事情：

- 申请 Android 麦克风权限。
- 实现按住说话或点击录音。
- 保存或上传录音。
- 接入语音识别服务。
- 将识别文本显示在聊天框。
- 根据识别内容触发 Luna 回复。

可选路线：

- 先使用云端语音识别。
- 后续再研究本地语音识别。

验收标准：

- 手机可以录音。
- 录音可以转成文字。
- 识别结果能进入对话流程。
- 失败时有明确提示。

## 阶段 6：v0.7 对话模型、人设和记忆

目标：让 Luna 开始具备基本陪伴对话能力。

要做的事情：

- 把 `character.md` 转成系统提示词。
- 建立基础人设配置文件。
- 接入一个对话模型 API。
- 根据用户输入生成回复。
- 回复时播放 TTS。
- 回复阶段映射到视频状态，例如 `thinking`、`speaking`、`happy`、`comforting`。
- 建立简单记忆，例如宝宝昵称、兴趣、近期话题。

早期记忆可以使用：

- 本地 JSON。
- SQLite。
- 轻量 key-value 存储。

验收标准：

- Luna 能按人设回复。
- 回复语气温柔、简短、有陪伴感。
- 可以记住少量基础信息。
- 视频状态、语音和伪口型能配合对话流程。

## 阶段 7：v0.8 iOS 自用版本

目标：在 iPhone 上验证同一套 App。

要做的事情：

- 配置 iOS Capacitor 项目。
- 使用 Xcode 真机运行。
- 配置麦克风权限。
- 测试 WebView 中视频 loop 播放性能。
- 测试音频播放和状态切换。

注意事项：

- iOS 自用安装比 Android 麻烦。
- 免费 Apple ID 可能有签名时间限制。
- 长期自用可能需要 Apple Developer 账号或 TestFlight。

验收标准：

- iPhone 可以运行 Luna App。
- 麦克风和音频播放正常。
- 视频播放性能可接受。

## 阶段 8：v0.9 树莓派屏幕版本

目标：把 Luna 迁移到树莓派和自有屏幕上。

要做的事情：

- 准备树莓派系统和屏幕。
- 使用 Chromium 全屏打开 Web 版本。
- 测试视频 loop 播放性能。
- 连接麦克风和扬声器。
- 测试语音输入、TTS 输出和状态切换。
- 配置开机自启动。

验收标准：

- 树莓派开机后自动显示 Luna。
- 可以播放语音并驱动口型。
- 基础交互稳定。
- 长时间运行不会明显卡顿或崩溃。

## 长期可选：Live2D 升级

目标：当视频 loop 方案验证出真实需求后，再评估是否升级到 Live2D。

触发条件：

- 视频素材维护成本过高。
- 需要更强实时互动。
- 需要更自然的表情、视线、身体动作和口型。
- 有预算找画师和绑定师，或找到合适的现成模型。

当前结论：

- 不作为 v0.2 到 v0.6 的主路线。
- 保留 `live2d-plan.md` 作为长期参考。

## 当前下一步

最建议马上做：

1. 整理当前已有视频素材。
2. 定义第一版状态机：`idle`、`listening`、`thinking`、`speaking`、`happy`、`comforting`。
3. 生成或补齐每个状态至少 1 条视频 loop。
4. 初始化 Android 混合 App 原型。
5. 在 App 中先跑通视频播放和状态切换。
