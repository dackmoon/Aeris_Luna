# Luna 网页验证方案 v0.1

本文档用于定义第一版网页版验证目标。v0.1 的网页不是完整产品，而是用于验证“Live2D 角色是否能在安卓手机网页中稳定显示、播放语音时是否能自动口型同步、后续是否适合迁移到树莓派屏幕”。

## 验证目标

第一版网页 Demo 需要回答几个问题：

- Luna 的形象在安卓手机屏幕上是否清晰、亲切。
- 用户是否能直观看懂她是一个陪伴助手。
- 现成 Live2D 模型是否能支撑陪伴助手的动态表现。
- TTS 或本地音频是否能驱动嘴巴开合。
- 对话框和角色展示是否适合宝宝/家长共同使用。
- 后续是否方便迁移到树莓派连接的小屏幕。

## v0.1 网页形态

建议先做一个竖屏优先的移动网页。

页面结构：

1. 顶部：Luna 名字和一句状态文案。
2. 中间：Live2D 角色画布。
3. 状态区：显示当前状态，例如待机、倾听、思考、说话。
4. 对话区：展示用户和 Luna 的聊天内容。
5. 底部：输入框、发送按钮和语音播放测试按钮。

v0.1 可以先用现成 Live2D 模型和本地音频测试，不急着接真实大模型接口和真实 TTS 服务。

## 安卓手机优先要求

第一版先针对安卓手机浏览器验证，尤其是 Chrome。

建议约束：

- 页面宽度按 `360px - 430px` 常见安卓屏幕适配。
- 竖屏体验优先。
- 按钮和输入框要足够大，方便触摸。
- Live2D 模型复杂度不要太高，避免安卓手机卡顿。
- 角色脸部、嘴巴和表情在小屏幕上必须清楚。
- 背景颜色低刺激，避免强对比。
- 不依赖鼠标悬停等桌面交互。
- 音频播放需要由用户点击触发，避免浏览器自动播放限制。

## 第一版交互

v0.1 建议先做最小交互：

- 用户输入一句话。
- 页面显示用户消息。
- Luna 返回一条预设或模拟回复。
- 播放一段本地音频或 TTS 音频。
- 播放音频时，使用音量驱动 Live2D 嘴巴参数。
- 根据回复类型切换 Live2D 表情或状态。

示例规则：

- 用户说“你好”：Luna 使用默认微笑。
- 用户说“我开心”：Luna 使用开心鼓励。
- 用户说“我难过”：Luna 使用安抚关心。
- 用户提问“为什么”：Luna 使用思考好奇。
- 用户说一长段话：Luna 使用温柔倾听。

这套规则足够验证第一版动态角色和口型同步，不需要一开始就做复杂 AI。

## 技术建议

如果进入代码实现，建议使用前端技术栈：

- 框架：React 或 Next.js。
- 样式：CSS Modules、Tailwind CSS 或普通 CSS。
- 角色：现成已绑定 Live2D 模型。
- Live2D：Live2D Cubism Web SDK 或成熟的 Web Live2D 渲染库。
- 音频：Web Audio API。
- 口型：使用音频音量驱动 `ParamMouthOpenY`。
- 数据：v0.1 使用本地 JSON 配置即可。

建议先不要在 v0.1 引入：

- 复杂后端。
- 登录系统。
- 数据库。
- 语音识别。
- 长期记忆。
- 从零制作 Luna 专属 Live2D 分层模型。

这些功能可以放到后续版本。

## 未来树莓派方向

长期如果要运行在树莓派和自有屏幕上，前期网页方案依然有价值，因为树莓派可以直接用浏览器全屏运行网页。

未来可能的实现方式：

- 树莓派启动后自动打开 Chromium 全屏。
- 本地网页显示 Live2D 角色和对话界面。
- 麦克风采集语音，后续接入语音识别。
- 扬声器播放 Luna 的 TTS 语音。
- 屏幕显示 Live2D 表情、待机动画、说话口型和对话状态。
- 本地保存基础记忆，必要时再同步云端。

因此 v0.1 网页应尽量保持轻量、响应式、可离线化，为未来设备端运行留空间。

## v0.1 验收标准

当满足以下条件时，可以认为 v0.1 验证完成：

- 有一版明确的 Luna 主形象方向。
- 有一个可测试的现成 Live2D 模型。
- 安卓手机网页能加载模型。
- 播放音频时嘴巴能跟随音量开合。
- 至少能切换 2 个状态或表情。
- 能说明后续如何从网页迁移到树莓派屏幕。
- 家长视角和宝宝视角的第一印象没有明显问题。
