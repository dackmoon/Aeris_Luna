# 版本约定

本文档说明 Aeris / Luna 项目的版本号含义、命名约定和发版流程。

## 命名层次

- **`Aeris`**：长期项目/体系名称。未来可以承载多个角色或多个产品形态。
- **`Luna`**：当前体系下的具体角色。当前阶段的代码原型也以 Luna 命名。
- 仓库名仍保留为 `Aeris_Luna`。如果未来出现 `Aeris` 下的其他角色，可以新建独立仓库或在本仓库分目录承载。

## 版本号

版本号格式：`vX.Y`。例如 `v0.1`、`v0.2`、`v1.0`。

**这不是 SemVer**。语义如下：

- **`v0.X`**：原型/迭代阶段。每个 `v0.X` 是一个实现阶段的里程碑。
- **`v1.0`**：第一个能稳定自用的完整版本。
- **`v1.X`** 之后再考虑是否引入 SemVer。

每个版本号的实际含义记录在 [CHANGELOG.md](../CHANGELOG.md)。

## Git tag 约定

每个版本里程碑打一个 git tag：

```bash
git tag v0.1 <对应 commit>
git tag v0.2 <对应 commit>
git push --tags
```

回看某个版本当时的状态：

```bash
git checkout v0.1
```

打 tag 的时机：

- 该版本的验收标准基本满足。
- 文档 (`CHANGELOG.md`) 已写好对应的版本条目。
- 已合并到 `main` 分支（如果走 PR 流程的话）。

## 不使用 archive/ 文件夹

历史快照统一用 git tag，不维护 `archive/v0.1/` 这种重复目录。理由：

- git 本身就保留完整历史，tag 即快照。
- `archive/` 会和 `docs/`、`assets/` 内容重复，长期维护成本高。
- 想看旧版本：`git checkout v0.X` 或在 GitHub 切换 tag 浏览。

## 分支约定

当前为单人自用项目，分支策略保持简单：

- **`main`**：稳定分支，每个 `v0.X` tag 都在 main 上。
- **`dev`**：日常开发分支，所有未发版的提交都在这里。
- 准备发新版本时，把 `dev` 合到 `main`，在 `main` 上打 tag。

未来若引入多人协作，再考虑 feature 分支和 PR 流程。

## 发版流程清单

发布一个新的 `v0.X`：

1. 在 `dev` 分支上完成所有阶段任务。
2. 更新 `CHANGELOG.md`，新增 `[v0.X]` 段落。
3. 检查 `docs/plan.md` 阶段状态是否已更新。
4. 把 `dev` 合并到 `main`。
5. 在 `main` 的对应 commit 上打 tag：`git tag v0.X`。
6. `git push origin main --tags`。
